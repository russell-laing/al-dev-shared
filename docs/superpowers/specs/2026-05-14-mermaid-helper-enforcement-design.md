# Mermaid Helper Enforcement Design

**Date:** 2026-05-14
**Status:** Approved

## Problem

`profile-al-dev-shared/markdown/md-mermaid-helper.md` contains mermaid best
practices (color palette, shape rules, error checklist) that are systematically
ignored when diagrams are generated by al-dev skills and agents. Users must
manually refer to the file each session.

Two independent gaps cause this:

1. **`applyTo` mismatch** — the frontmatter restricts auto-injection to
   `**/*.mmd` (standalone mermaid files). All mermaid in this project is
   embedded inline in `.md` files, so auto-injection never fires.

2. **Cold-start agents** — subagents dispatched by skills start with no
   inherited context. They only see their system prompt and dispatch prompt.
   The docs-writer says "follow `markdown/md-mermaid-helper.md`" but never
   reads it; the architect and support agents don't reference it at all.

## Chosen Approach

Fix both gaps surgically without duplicating content or adding new abstraction:

- Expand `applyTo` in `md-mermaid-helper.md` to cover `.md` files.
- Add explicit read steps to the three agents that produce diagrams.

## Changes

### Change 1 — Expand `applyTo` in `md-mermaid-helper.md`

**File:** `profile-al-dev-shared/markdown/md-mermaid-helper.md`

Change frontmatter from:

```yaml
applyTo: "**/*.mmd"
```

to:

```yaml
applyTo: "**/*.{md,mmd}"
```

This causes the harness to auto-inject the mermaid rules into the main
Claude Code session whenever any `.md` file is active — covering the vast
majority of diagram work done outside subagents.

### Change 2 — Explicit read steps in diagram-producing agents

All three agents use the `AL_DEV_SHARED_PLUGIN_ROOT` token (defined in
`knowledge/harness-concepts.md`) rather than a hardcoded path.

#### `al-dev-docs-writer.md`

The existing `## Diagram Rules` section says "Follow
`markdown/md-mermaid-helper.md`" without reading the file. Replace that
reference with an explicit instruction:

> Before generating any mermaid diagram, read
> `$AL_DEV_SHARED_PLUGIN_ROOT/markdown/md-mermaid-helper.md`
> and apply all rules from it.

#### `al-dev-solution-architect.md`

Diagrams are allowed for MEDIUM/COMPLEX plans but there is no mermaid
guidance. Add the same explicit read instruction alongside the diagram
permission note.

#### `al-dev-support-agent.md`

No diagram provision currently exists. Add a note: if the agent judges a
diagram would aid the customer reply, it must first read
`$AL_DEV_SHARED_PLUGIN_ROOT/markdown/md-mermaid-helper.md` before
generating it.

## Scope

Files changed:

| File | Change |
| --- | --- |
| `profile-al-dev-shared/markdown/md-mermaid-helper.md` | `applyTo` frontmatter |
| `profile-al-dev-shared/agents/al-dev-docs-writer.md` | Diagram Rules section |
| `profile-al-dev-shared/agents/al-dev-solution-architect.md` | Diagram section |
| `profile-al-dev-shared/agents/al-dev-support-agent.md` | Diagram note added |

## Out of Scope

- Skills that do not produce diagrams directly
- Creating a wrapper skill for diagram generation
- Embedding mermaid rules inline in `md-helper.md`

## Success Criteria

- Main session auto-injects mermaid rules when any `.md` file is active
- Any agent that generates a diagram has read the helper before doing so
- No content duplication between `md-helper.md` and `md-mermaid-helper.md`
- No new abstraction layers or wrapper skills introduced

# Review Docs

## Purpose

`docs/` contains human-authored reference files alongside auto-generated maps and
quality reports. The human-authored files (architecture decisions, development commands,
naming conventions, etc.) can drift from the codebase: scripts get renamed, skills get
archived, file paths change.

This skill surfaces that drift and readability issues before they mislead future readers.

## Phase-proof requirement

This skill follows `../../knowledge/phase-proof-contract.md` — emit a phase-proof block at each phase boundary before reporting completion or updating `.dev/health-loop-state.md`.

## When NOT to Use

- **Auto-generated files** (`al-dev-*-map.md`, `workflow_diagrams.md`, `plugin_graph.md`) — Phase 1 excludes these automatically
- **Audit report outputs** (`al-dev-*-quality.md`) — regenerated by skills, not maintained by hand
- **Files in subdirectories** (`docs/health/`, `docs/example_runs/`) — Phase 1 excludes these; use dedicated audits for those surfaces
- **Mermaid diagram or documentation-map accuracy** — use `/sync-map-documentation`; this skill checks prose and references, not map nodes, generated catalog sections, or diagram edges

## Phase 0: Parse Arguments

Read `$ARGUMENTS`:

- `--file <filename>` → set `TARGET_FILE=filename`, restrict review to that file only
- `--fix` → set `FIX_MODE=true`, offer targeted edits after reporting

## Phase 1: Identify Human-Authored Review Targets

```bash
# All .md files directly in docs/ (not subdirectories)
find docs -maxdepth 1 -name "*.md" | sort

# Files with actual BEGIN GENERATED sections (auto-generated content)
grep -rl "<!-- BEGIN GENERATED:" docs/*.md 2>/dev/null

# Quality-report outputs (named al-dev-*-quality.md)
ls docs/al-dev-*-quality.md 2>/dev/null
```

Build the review list by excluding:

- Files that contain `<!-- BEGIN GENERATED:` as an actual HTML comment (auto-generated
  sections) — note: a file that merely *mentions* the pattern in prose is NOT excluded
- Files matching `al-dev-*-quality.md` (audit report outputs, regenerated by skills)

If `TARGET_FILE` is set, restrict to that file only (error if it does not exist).

Print: `Reviewing N files in docs/ (skipping M auto-generated/report files)`

## Phase 2: Review Each File

Phase 2 owns the per-file loop. For each file in the review list from Phase 1, set it as
the current file (`<file>`) and run Phase 3 (Technical Accuracy) then Phase 4
(Readability) against that one file. Phases 3 and 4 do not loop themselves — they operate
on the current file only. Record all findings for every file before moving to Phase 5.

## Phase 3: Technical Accuracy

Run these checks against the current file (`<file>`) selected by the Phase 2 loop. Do not
re-iterate the review list here.

### Verification procedure

Use one verification shape for every reference you extract:

- Resolve the concrete path from the current file.
- Apply the matching adapter below.
- Record behavior, not line counts: exists and resolves cleanly, missing, or
  synthetic/example path skipped by design.

**Script references** — verify every `scripts/*.py` or `scripts/*.sh` path exists:

```bash
grep -oE 'scripts/[a-zA-Z0-9_.-]+\.(py|sh)' docs/<file> | sort -u
```

For each match:

```bash
test -f <path> && echo "✅ <path>" || echo "BROKEN: <path>"
```

**Skill references** — verify every skill invocation (backtick-wrapped prose and bare
code-block lines) exists on disk:

```bash
# Catches both `/skill-name` in prose and `/skill-name [--args]` at line start in code blocks
grep -oE '`/[a-z][a-z0-9-]+`|^\s*/[a-z][a-z0-9-]+' docs/<file> | \
  grep -oE '/[a-z][a-z0-9-]+' | sort -u
```

For each `/skill-name` in the list, strip the leading `/` to get `<skill-name>` and check
both skill surfaces:

```bash
{ test -d "profile-al-dev-shared/skills/<skill-name>" || \
  test -d ".claude/skills/<skill-name>"; } && \
  echo "✅ /<skill-name>" || echo "BROKEN: /<skill-name>"
```

**Archived path references** — any path containing `archived/` should still exist:

```bash
grep -oE '[a-zA-Z0-9/_.-]*archived/[a-zA-Z0-9/_.-]+' docs/<file>
```

For each concrete path (not a prose example), verify:

```bash
test -d <path> || test -f <path> || echo "MISSING archived: <path>"
```

**Key project-relative paths** — find backtick-wrapped paths that look like real
project files:

```bash
grep -oE '`[a-zA-Z][a-zA-Z0-9_/.-]+\.(md|json|py|sh|yml)`' docs/<file>
grep -oE '`[a-zA-Z][a-zA-Z0-9_/-]+/`' docs/<file>
```

Verify all results if ≤10 total; otherwise spot-check the first 10 non-synthetic ones.
Skip clearly synthetic examples (`{surface}/`, `<name>.md`) and user-relative paths
(`~/.claude/`). Use `test -f` / `test -d` to verify each.

## Phase 4: Readability

Run these checks against the current file (`<file>`) selected by the Phase 2 loop. Do not
re-iterate the review list here.

**Code blocks without language tags:**

```bash
grep -n '```$' docs/<file>
```

Each bare `` ``` `` line is either a closing fence (follows a tagged opening) or an
untagged opening. Scan the results in line-number order — a bare fence that opens a
block rather than closes one is a finding. Flag each such line number.

**Thin sections** — a `##` heading with fewer than 2 non-blank body lines before the
next heading is potentially a stub:

```bash
awk '/^## /{if(hdr && body<2) print "THIN:", hdr; hdr=$0; body=0; next}
     /^[[:space:]]*$/{next}
     hdr{body++}
     END{if(hdr && body<2) print "THIN:", hdr}' docs/<file>
```

Flag any heading that prints. Headings with intentionally minimal content (like a
`## Phase 0` with a single bullet) may be false positives — use judgment.

**Stale "Last updated" date:**

```bash
grep "Last updated" docs/<file>
```

Dates more than 6 months before today are potentially stale.
Flag as `[STALE?]` with the actual date.

## Phase 5: Write Findings Report

After reviewing all files, write to `docs/docs_quality.md`:

```markdown
# Docs Quality Review

Generated: [today's date]
Files reviewed: N | Skipped (auto-generated/reports): M

## Summary

- BROKEN references (accuracy): N
- Readability/staleness warnings: N

---

## <filename>

**Last updated:** <date from file, or "no date marker">

### Technical Accuracy

| Type | Finding | Detail |
|------|---------|--------|
| ✅ scripts | all N verified | — |
| BROKEN | `scripts/foo.py` | File not found on disk |
| BROKEN | `/old-skill` | No matching skill in either skill surface |

### Readability

| Type | Finding | Detail |
|------|---------|--------|
| WARN | Line 42 | Code block missing language tag |
| WARN | `## Section Name` | 0 content lines |
| STALE? | Last updated 2025-10-01 | More than 6 months ago |

---
```

Repeat the per-file block for each reviewed file. Append a `## Skipped` section listing
the auto-generated and report files that were excluded, with a one-line reason each.

Verify the report was written:

```bash
ls -la docs/docs_quality.md
```

## Phase 6: Report and Offer Fixes

Print a brief summary:

```text
Docs review complete. Report: docs/docs_quality.md
BROKEN references: N
Readability warnings: N
```

List any BROKEN findings by file.

If `FIX_MODE=true` or the user asks to fix issues:

- **Broken script path** → `find scripts/ -name "*<basename>*"` to find the renamed
  script; suggest the corrected path or note the script was removed
- **Broken skill reference** → check `profile-al-dev-shared/archived/skills/<name>/`
  and `.claude/skills/`; suggest replacement or note it was archived
- **Missing language tag** → offer the `Edit` for each affected code fence
- **Thin section** → show the heading and ask what content is expected; do not
  auto-fill without user input
- **STALE?** date → ask if the file is still current; update the date if confirmed

Do not auto-apply fixes without user confirmation.

## Success Criteria

✅ All broken script and skill references identified  
✅ Missing code-block language tags flagged with line numbers  
✅ Stale "Last updated" dates surfaced  
✅ Auto-generated and report files excluded from review  
✅ Report written to `docs/docs_quality.md` and verified on disk
